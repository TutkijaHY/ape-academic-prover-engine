01 — Overview

A research colleague, not a chatbot

APE lets researchers pose questions directly against their own corpus — published scholarly literature, self-authored research texts, and numerical data sources — and receive responses where every claim is automatically traced to its original document.

The strategic value is not merely that APE uses language models. It is that corpus querying, source-aware reasoning, manuscript assessment, citation-focused analysis, and teaching-oriented output generation already exist within one coherent architecture. For an acquirer, this is closer to a research software asset than a single-purpose demo.

The system is best understood as a set of research workflows rather than a single chat feature. These workflows cover daily corpus use, manuscript diagnostics, evidence verification, external literature extension, and teaching output generation.

Query
Corpus-grounded questioning. Three response styles: Prover (claim-by-claim), Analyst (verbalized sampling, based on Stanford research), and Brief.
MODE 02

06 — Design Decisions

Why the system is more than a thin wrapper around an LLM

HHI Source Diversity as Retrieval Constraint
The Herfindahl-Hirschman Index applied at retrieval level. If one source dominates (HHI ≥ 0.50), Self-RAG triggers re-retrieval. APE ranks by relevance and epistemic diversity — not relevance alone.
Three-Tier Source Typing [S]/[R]/[D]
APE separates the corpus into three layers: [S] Scholarly articles, [R] Own research texts, and [D] Numerical data. Each layer uses different chunking, retrieval weighting, and diversity calculation — answers are drawn from the right type of evidence.
Unified Multimodal Index
APE indexes both text and images from PDF documents. A Vision Language Model generates descriptions of images and figures; tables are parsed at structural level using IBM Granite-Docling (TEDS-structure 0.97, IBM-reported). One query retrieves prose, tables, and visual evidence from the same index.
Configurable Self-RAG Quality Gate
Before delivering an answer, APE checks three criteria: enough sources found, no single source dominating ([S]/[R]/[D] balance maintained), and source diversity within acceptable limits. If any criterion fails, retrieval runs again. All thresholds set in .env.
Model-Agnostic by Design
APE has no hardcoded LLM model names anywhere in the codebase. The active model is selected from a dropdown list of locally installed Ollama models. Switching to a new LLM requires no code changes — token limits estimated dynamically. A built-in comparison view runs the same query across all locally installed models simultaneously.
100% Local Processing
All processing runs on the local machine — no cloud dependencies, no API keys, no data transmitted externally. Critical for research institutions, universities, law firms, and any organisation handling sensitive or unpublished material.
